Agristability Enrollment – Action Security Implementation Plan

Cross-app, Dataverse-native security model  |  Prepared April 30, 2026

Executive Summary

The Agristability Enrollment app currently identifies four user types at runtime via Dataverse role and team lookups. This plan describes how to move action-level security enforcement outside the application layer into Dataverse itself, so the same rules apply to any app, flow, or API that touches the same data — not just the enrollment Power Apps canvas app.

The recommended approach combines three Dataverse-native mechanisms:

  1. Security Roles — restrict which entity rows and columns each user type can read/write.
  2. Custom APIs — server-side entry points for each major action that check caller permissions before executing.
  3. Plug-in Steps — pre-operation enforcement that blocks disallowed direct-OData writes regardless of the calling app.
Guiding principle: No action-level rule should live only in a client app. If a rule can be bypassed by calling OData directly, it is not secure.

1
Audit & Document Existing Security Roles

Goal

Establish a baseline of what Dataverse security roles and team memberships currently grant.

Tasks

  1. Export the current System Administrator and Enrolment Admin role privilege lists from Power Platform Admin Center.
  2. Document which entities each role has Create/Read/Update/Delete (CRUD) access to — specifically vsi_participantprogramyears, queueitems, queues, systemusers.
  3. Confirm that Supervisor Approval Queue and Verifier Team have defined membership processes (who adds/removes members, and how).
  4. Identify any privilege gaps or over-provisioning relative to the Action Permission Matrix above.

Deliverable

A spreadsheet mapping each user type to current Dataverse entity privileges, highlighting gaps vs. the target matrix.

2
Define & Tighten Dataverse Security Roles

Goal

Ensure each role grants only the entity-level access required. This is the first layer of defense and applies to all apps, flows, and direct API calls automatically.

Tasks

  1. Enrolment Admin role — Grant Write on vsi_participantprogramyears for the fields it needs (e.g., vsi_taskstatus, vsi_enrolmentstatus, ownerid). Deny direct write on vsi_enrolmentstatus for values that should only be set via Custom API (see Phase 3).
  2. Queue/Verifier roles — Create dedicated security roles (e.g., Enrolment Verifier, Enrolment Supervisor) with Read-only on the main entity and Write only on the specific fields each action requires.
  3. Field-level security profiles — Use Dataverse Column Security Profiles to restrict sensitive calculated fields (e.g., vsi_calculatedenfee, vsi_variancecalculation) so Verifiers cannot write them directly.
  4. Assign the correct base security role to each Azure AD group that maps to a user type, so membership changes in Azure AD / Entra ID automatically propagate to Dataverse.

Deliverable

Updated security roles published in the target environment, with a before/after privilege diff recorded.

3
Create Dataverse Custom APIs for Each Action

Goal

Replace direct OData PATCH/DELETE calls in client apps with server-side Custom API calls. Each Custom API owns its own authorization check and business logic, making it reusable by any app (Canvas, Model-driven, Power Automate, external API).

Custom APIs to Create

API Unique Name Action Covered Minimum Required Role Returns
vsi_AssignWorker Assign enrolment owner, delete queue item Queue Member Success / error message
vsi_ApproveFees Set task status = Approved, stamp approved date Enrolment Admin Success / error message
vsi_ReferToSupervisor Set task status = PendingSupervisor, add to queue Verifier (any user) Success / error message
vsi_MarkManual Set task status = Manual, set enrolment status = ToBeReviewed Queue Member Success / error message
vsi_SendBulkNotices Trigger notice workflow for selected enrolments Enrolment Admin Count sent / errors

Each Custom API plug-in should

  1. Retrieve the calling user's security roles and team/queue memberships.
  2. Evaluate against the permission matrix — throw InvalidPluginExecutionException with a user-friendly message if unauthorized.
  3. Execute the business logic (field updates, queue operations, workflow triggers).
  4. Return a structured response (success flag + optional message).

Implementation Notes

  • Custom APIs are registered in the Dataverse solution and deployed alongside entity customizations — no separate infrastructure needed.
  • The plug-in assembly can be shared by multiple Custom APIs, with each action as a separate plug-in step class.
  • Custom APIs support both synchronous and async execution; most of these actions should be synchronous.
  • Client apps call them via POST /api/data/v9.2/vsi_AssignWorker (unbound) or bound to a record as needed.
Cross-app benefit: Once deployed, a Model-driven app, a Power Automate flow, or an external integration calling the same Custom API will automatically get the same security enforcement — no duplication of logic.
4
Plug-in Pre-Operation Guards (Defense in Depth)

Goal

Even if a caller bypasses the Custom API and calls OData directly, a pre-operation plug-in on the entity can reject disallowed writes. This is the last line of defense.

Tasks

  1. Register a pre-operation Update plug-in on vsi_participantprogramyears. For each write, inspect the target image fields:
    • If vsi_enrolmentstatus is being changed, verify the caller holds at least the Queue Member role or higher.
    • If calculated fee fields are being changed, verify the caller holds Enrolment Admin or higher.
    • Block and return a descriptive error message for any violation.
  2. Register a pre-operation Delete plug-in on queueitems to ensure only authorized users can remove items from the Supervisor Approval Queue.

Implementation Notes

  • Pre-operation plug-ins fire in the database transaction — rejection rolls back the entire operation cleanly.
  • Keep the role-check logic in a shared helper class so Custom API plug-ins and guard plug-ins stay in sync.
  • Use the IOrganizationService in the plug-in context to retrieve roles — avoid hard-coding role GUIDs; look them up by name and cache in static state.
5
Update Client Apps to Call Custom APIs

Goal

Replace direct service calls in the enrollment app (and any other apps) with calls to the new Custom APIs.

Enrollment App Changes

Current Call Replace With
QueueitemsService.delete() + Vsi_participantprogramyearsService.patch() (assign) POST vsi_AssignWorker
Vsi_participantprogramyearsService.patch(vsi_taskstatus: Approved) POST vsi_ApproveFees
Vsi_participantprogramyearsService.patch(vsi_taskstatus: PendingSupervisor) POST vsi_ReferToSupervisor
Vsi_participantprogramyearsService.patch(vsi_taskstatus: Manual) POST vsi_MarkManual

UI Behavior

  • The app should still show or hide action buttons based on the user's resolved role (current behavior), providing a good UX.
  • However, the server will enforce the rule independently — a hidden button is UX, not security.
  • On a 403 / plug-in error response, display a toast: "You don't have permission to perform this action."

7
Deployment & Governance

Solution Packaging

  • Package all security roles, Custom API registrations, plug-in assembly, and column security profiles in a single Dataverse managed solution (e.g., AgristabilityActionSecurity).
  • Managed solution prevents accidental edits in target environments.
  • Deploy via standard ALM pipeline (dev → test → prod).

Governance Notes

  • Role assignment changes should go through a change-management process, not ad-hoc admin edits.
  • Queue and team memberships are the source of truth for Queue Member and Verifier access; keep them clean.
  • Any new action added to any app must be added to the Custom API layer first — client apps are downstream consumers.
  • Log Custom API calls using Dataverse audit logging for accountability.
